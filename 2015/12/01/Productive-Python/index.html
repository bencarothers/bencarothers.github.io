Productive Python

A few tools that keep me going

Posted by Ben Carothers on December 1, 2015

I’ve Fallen For Python.

It’s no secret that I’m a fan of the language. Python has become my go to too for small weekend projects and my assignments at school – especially in cases where the scope and complexity of the problem isn’t necessarily a limiting factor.

it’s fun to use, the code generally ends up being clean and concise, and it’s really useful for fast, iterative development. Furthermore, the huge number of awesome libraries make it possible to do nearly anything your’re willing to research.

I’ve recently come across a few tools that, not all necessarily linked to Python, that have made working in the language both more pleasant and also productive.

Pyenv and Virtualenv

Dealing with different versions of Python and also different versions of packages within these versions can be frustrating. If you work in Python for very long without using virtual environments you’ll undoubtedly end up with some conflicting versions and a lengthy output from pip freeze.

I’ve recognized the benefits of virtual environments for awhile, but I could never commit to using them regularly because it always felt so clunky. Since the projects I’m currently involved in are using entirely different versions of Python I finally commited to finding a set up that worked for me.

Pyenv is an awesome tool that lets you seemlessly download and switch between different versions of Python. Furthermore, by issuing the command, pyenv local /version/, you can assign a version of Python to any specific directory.

By then using the plugin Pyenv-virtualenv you can create virtual environments for specific projects using the installed version. For instance, I can create a virtual environment for my blog by typing pyenv virtualenv 2.7.9 blog. Then any packages I install will only exist in blog once I type pyenv activate blog.

Encapsulating environments makes testing and coordinating with other team members much easier.

Setuptools and Travis

I’ve also been using Travis as a contiuous integration service. As you’re developing something is going to break, and it won’t always be something that you’re expecting. In fact you usally won’t expect it at all.

Using Setuptools you can create a setup.py file used to create a distrubtable and run the tests-suite after locating and downloading all the dependencies.

Then you can register your Github repository with Travis, create a small .yaml file that builds your project and tests it every time you push code. This has been wonderful because the functionality youre working on isn’t always what breaks, so you know where to look without having to manually run your tests. It often tells me there’s a bug long before I would’ve noticed it during development.

Coveralls

Lastly, I’ve been using Coveralls.io with the package coverage and in cooperation with Travis to get an idea of where my tests aren’t reaching. Coveralls gives you a line by line audit of you’re code showing you which parts are covered in your tests and where your code is bare.

Obviously, a high percentange doesn’t necessarily mean that every case is handled, but the output does give you an idea of areas that could use some checks.

If nothing the button on my README keeps me accountable, which is useful so that I don’t code blindly for hours only to realize I’ve broken some core functionality.

There are a ton of tools that make development more productive and fun. These are just a few of the ones that I’ve been using lately. Check them out here for more info:

  • Setuptools
  • Pyenv
  • Travis
  • Coveralls
